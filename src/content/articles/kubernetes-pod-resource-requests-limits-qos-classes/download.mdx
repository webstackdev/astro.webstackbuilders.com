Every pod in Kubernetes consumes CPU and memory, but how you declare those resources determines where pods land, how they behave under pressure, and which pods die when nodes run low. Requests and limits aren't optionalâ€”they're the contract between your workload and the cluster. A production incident saw random pod evictions at 3 AM: 60% of pods had no memory limits, a few memory hogs consumed all RAM, and the well-behaved pods with requests but no limits were evicted first because they were marked "Burstable" QoS.

QoS class determines eviction order, but most teams don't realize what class their pods belong to until they're debugging an outage. Understanding the resource model and QoS classification is essential for reliability.
